# 개발 하네스 작업 계약

이 문서는 이 저장소에서 사람과 코딩 에이전트가 공통으로 따라야 하는 기본 작업 계약을 정의한다.

목표는 다음 네 가지다.

- 요구사항과 성공 기준을 먼저 고정한다.
- 구현 변경이 저장소 구조와 충돌하지 않게 한다.
- 테스트와 문서 동기화를 기본 경로에 포함한다.
- 커밋 직전 게이트를 일관되게 적용한다.

## 공식 명령

### `/HARNESS-STRICT` 또는 `./HARNESS-STRICT`

`/HARNESS-STRICT`와 `./HARNESS-STRICT`는 이 저장소의 공식 하네스 강제 명령이다.

이 명령은 새로운 규칙을 만드는 것이 아니라, 이 문서의 기존 하네스 규칙을 작업 시작 시점부터 엄격하게 적용하라는 트리거다.

명령이 포함된 요청은 아래 효과를 가진다.

- 작업 파일을 먼저 작성한다.
- 모호한 요구사항은 질문 후 대기한다.
- 가정이 있으면 승인 후 진행한다.
- 구현 계획을 설명하고 승인받기 전에는 코드 작성하지 않는다.
- 구현 단계 상태와 승인된 파일 목록을 작업 파일에 먼저 기록한다.
- 구현 후 기본 검증을 통과해야 한다.

## 기본 원칙

- 작업 시작 전에는 `docs/harness/task-template.md` 형식으로 작업 의도를 정리한다.
- 로컬 작업 스펙 파일의 기본 위치는 `plan/tasks/*.md`다.
- 하네스는 기본적으로 `plan/tasks/` 아래에서 가장 최근에 수정된 Markdown 파일을 현재 작업 스펙으로 사용한다.
- 특정 작업 파일을 강제로 지정하려면 `TASK_FILE=plan/tasks/<file>.md`로 실행한다.
- 요구사항이 모호하면 구현 전에 반드시 사용자에게 추가 정보를 요청한다.
- 요구사항이 부족해 에이전트가 임의 판단을 해야 하면, 그 판단을 작업 파일에 기록하고 반드시 사용자 승인을 받은 뒤 진행한다.
- 구현 계획을 세운 뒤에는 실제 코드를 작성하기 전에 어떤 파일, 함수, 메서드, 타입을 추가/수정할지 사용자에게 설명하고 승인을 받아야 한다.
- 커밋 메시지는 Conventional Commit 형식을 따르되, 요약은 한글로 작성한다.
- 구조, API, 책임 경계가 바뀌면 관련 `docs/` 문서도 함께 갱신한다.
- 하네스 문서나 검증 로직이 비대해지거나 책임이 과도하게 늘어나면 관련 책임별로 분리한다.
- 커밋 전에는 반드시 `scripts/agent-check.sh fast`를 통과한다.
- 커밋은 가능하면 `scripts/agent-commit.sh`를 사용한다.

## 요구사항 명확성 규칙

- 작업 파일에는 요구사항 명확성 상태를 반드시 기록한다.
- 요구사항이 모호하면 `Requirements-Clarity: unclear`로 기록하고, 사용자에게 어떤 정보를 요청했는지 남긴다.
- 사용자 응답으로 모호성이 해소되기 전에는 구현과 커밋을 진행하지 않는다.
- 하네스는 `Clarification-Status: pending` 상태에서 검증을 통과시키지 않는다.

## 가정과 승인 규칙

- 요구사항 부족으로 임의 판단이 들어가면 그 내용을 작업 파일의 `가정 및 승인` 섹션에 기록한다.
- 가정이 없다면 `Assumptions-Used: no`로 기록한다.
- 가정이 있다면 `Assumptions-Used: yes`와 함께 사용자 승인 상태를 `Assumption-Approval: approved`로 명시해야 한다.
- `Assumption-Approval: pending` 또는 누락 상태에서는 구현을 진행하지 않는다.

## 구현 계획 승인 규칙

- 작업 파일에는 구현 전에 작성한 `구현 계획 및 승인` 섹션이 있어야 한다.
- 이 섹션에는 최소한 수정할 파일, 추가/수정할 함수나 메서드, 예상되는 공개 계약 변화, 테스트 계획을 적는다.
- 사용자가 계획을 확인해 승인하기 전에는 코드 작성과 커밋을 진행하지 않는다.
- 하네스는 `Implementation-Approval: approved`가 없으면 검증을 통과시키지 않는다.
- 하네스는 `Implementation-Step-Status: approved-for-implementation`가 없으면 구현 단계로 들어가지 않는다.
- 하네스는 `Approved-Files`에 없는 스테이징 파일이 있으면 검증을 통과시키지 않는다.

## 하네스 분리 규칙

- 하네스 규칙이나 검증 로직이 많아져 한 파일이 여러 책임을 동시에 가지기 시작하면 분리한다.
- 문서는 주제별 디렉토리와 파일로 유지하고, 한 문서가 여러 독립 규칙 집합을 모두 떠안지 않게 한다.
- 스크립트는 책임별로 쪼갠다. 예를 들어 요구사항 검증, 코드 스타일 검증, 문서 동기화 검증은 별도 스크립트나 모듈로 나눌 수 있어야 한다.
- 새로운 규칙을 추가할 때는 기존 파일에 단순 누적하지 말고, 책임 분리가 더 적절한지 먼저 검토한다.
- 하네스 자체를 수정할 때도 같은 코드 컨벤션과 승인 절차를 따른다.

## 변경 범위 규칙

- 구현 코드는 기존 패키지 책임을 따른다.
- 새로운 패키지나 디렉토리를 추가할 때는 먼저 기존 책임 분리를 재사용할 수 있는지 검토한다.
- `main.go`에는 wiring 이외의 상세 비즈니스 로직을 넣지 않는다.
- `internal/` 아래의 패키지 책임이 의미 있게 바뀌면 `docs/conventions/directory-convention.ko.md` 또는 `docs/architecture/architecture.ko.md`를 갱신한다.

## 코드 컨벤션

- 코드는 사용자가 빠르게 읽을 수 있도록 작게 유지한다.
- 새로운 함수나 메서드는 가능한 한 15줄을 넘기지 않는다.
- 15줄을 넘길 정도로 역할이 늘어나면 보조 함수로 분리한다.
- 예외가 꼭 필요하면 작업 파일에 이유를 남기고 사용자 승인을 받은 뒤 진행한다.
- 하네스는 스테이징된 Go 파일에 대해 함수 길이 제한을 검사한다.

## 테스트 규칙

- 동작 변경이 있으면 관련 테스트를 추가하거나 기존 테스트를 보강한다.
- 기본 검증 명령은 `scripts/agent-check.sh fast`다.
- 소켓 의존 테스트까지 확인해야 할 경우 `scripts/agent-check.sh full`을 사용한다.
- 포맷 검사는 자동 수정이 아니라 실패 보고를 우선한다.

## 커밋 메시지 규칙

- 형식은 Conventional Commit을 사용한다.
- 허용 타입은 `feat`, `fix`, `docs`, `refactor`, `test`, `chore`다.
- scope는 선택 사항이며 영문 식별자를 써도 된다.
- 콜론 뒤 요약은 한글로 작성한다.

예:

```text
feat(harness): 로컬 개발 하네스 추가
fix(dashboard): 정적 파일 경로 검증 오류 수정
docs(api): 대시보드 네임스페이스 API 설명 보강
```

## 문서 동기화 규칙

- `internal/admin`, `internal/dashboard` 변경은 `docs/api/dashboard-api.ko.md` 점검 대상이다.
- 구조나 책임 변화는 `docs/conventions/directory-convention.ko.md` 또는 `docs/architecture/architecture.ko.md` 점검 대상이다.
- 타입 의미나 용어 계약이 바뀌면 `docs/conventions/type-reference.ko.md` 점검 대상이다.
- 문서 수정이 불필요하다고 판단하면 현재 작업 스펙 파일에 이유를 남긴다.

권장 형식:

```md
## 문서 영향

- Docs-Impact: none
- Docs-Reason: 내부 리팩터링만 포함하며 공개 계약과 패키지 책임은 바뀌지 않음
```

## 금지 행동

- 검증 실패를 무시한 채 기본 경로로 커밋하지 않는다.
- 사용자 변경을 되돌리기 위해 무단으로 `git reset --hard` 같은 파괴적 명령을 쓰지 않는다.
- 문서 갱신이 필요한 변경을 코드만 수정한 채 끝내지 않는다.
- `internal/dashboard/static/index.html` 같은 필수 런타임 자산의 존재를 가정만 하고 넘어가지 않는다.
- 모호한 요구사항을 질문 없이 임의 해석해 구현하지 않는다.
- 사용자 승인 없이 가정을 사실처럼 확정하고 구현하지 않는다.
- 구현 전에 어떤 코드를 작성할지 설명하지 않은 채 바로 코드를 쓰지 않는다.
- 구현 계획에 대한 사용자 승인 없이 코드를 작성하지 않는다.
