# RPS 시나리오와 서비스 규모 해석

이 문서는 `300/600/900/1200 RPS` 같은 요청률을 서비스 규모나 운영 시나리오로 해석할 때 사용할 기준을 정리한다.

핵심 원칙은 단순하다.

- `RPS`는 사용자 수가 아니라 요청 도착률이다.
- 같은 `RPS`라도 요청 비용, 응답시간, 호출 패턴에 따라 실제 서비스 규모 의미가 크게 달라진다.
- 따라서 `RPS`는 항상 `요청 성격`, `호출 빈도`, `응답시간`, `엔드포인트 범위`와 함께 해석해야 한다.

## 먼저 구분할 것

같은 `1200 RPS`라도 아래는 의미가 다르다.

- 프록시 앞단의 가벼운 `GET` 요청
- 캐시 히트가 많은 읽기 API
- DB 조인과 권한 검사가 많은 조회 API
- 주문, 결제, 쓰기 중심 API
- 내부 마이크로서비스 간 fan-out 호출

따라서 "`RPS`만으로 서비스 규모를 단정"하면 해석이 왜곡된다.

## 계산의 기본식

### 1. 요청률과 사용자 활동

사용자 행동을 단순화하면 아래처럼 계산할 수 있다.

```text
RPS = 동시 활성 사용자 수 × 1인당 초당 요청 수
동시 활성 사용자 수 = RPS / 1인당 초당 요청 수
```

이 식은 요청률 기반 부하 생성 도구가 실제로는 초당 몇 개 요청을 시작할지 정의한다는 점과 맞닿아 있다.

### 2. 요청률과 동시 처리량

응답시간이 길수록 같은 `RPS`에서도 시스템에 걸리는 동시 처리량은 커진다.

```text
동시 처리 중 요청 수 ≈ RPS × 평균 응답시간(초)
```

이 식은 Little's Law를 단순 적용한 직관적 해석이다.

## 지속 요청량 환산

아래 값은 `24시간 지속`을 가정한 단순 환산이다.

| RPS | 하루 요청 수 | 30일 요청 수 |
| --- | ---: | ---: |
| `300` | 약 `25.9M` | 약 `777.6M` |
| `600` | 약 `51.8M` | 약 `1.56B` |
| `900` | 약 `77.8M` | 약 `2.33B` |
| `1200` | 약 `103.7M` | 약 `3.11B` |

이 수치는 실제로는 보통 "서비스 전체 평균"보다 "특정 엔드포인트 또는 피크 시간대" 해석에 더 유용하다.

## 사용자 행동 가정별 환산

아래 표는 모두 **추정치**다. 절대값이 아니라 행동 패턴에 따른 감을 잡기 위한 예시다.

### 가정 A: 사용자 1명이 평균 `10초`마다 1번 호출

- 1인당 초당 요청 수: `0.1`

| RPS | 추정 동시 활성 사용자 |
| --- | ---: |
| `300` | `3,000명` |
| `600` | `6,000명` |
| `900` | `9,000명` |
| `1200` | `12,000명` |

해석:

- 짧은 폴링, 자동 새로고침, 자주 보는 대시보드 조회 API에 가깝다.
- B2B SaaS의 핵심 읽기 API 기준으로는 이미 의미 있는 피크 부하다.

### 가정 B: 사용자 1명이 평균 `30초`마다 1번 호출

- 1인당 초당 요청 수: 약 `0.0333`

| RPS | 추정 동시 활성 사용자 |
| --- | ---: |
| `300` | 약 `9,000명` |
| `600` | 약 `18,000명` |
| `900` | 약 `27,000명` |
| `1200` | 약 `36,000명` |

해석:

- 일반적인 목록 조회, 페이지 전환, 약한 배경 갱신에 가까운 시나리오다.
- 이 경우 `1200 RPS`는 적지 않은 동시 활성 사용자 규모로 읽는다.

### 가정 C: 사용자 1명이 평균 `60초`마다 1번 호출

- 1인당 초당 요청 수: 약 `0.0167`

| RPS | 추정 동시 활성 사용자 |
| --- | ---: |
| `300` | 약 `18,000명` |
| `600` | 약 `36,000명` |
| `900` | 약 `54,000명` |
| `1200` | 약 `72,000명` |

해석:

- 호출 빈도가 낮은 조회 API, 백그라운드 갱신 비중이 작은 제품에 가까운 해석이다.
- 같은 `RPS`라도 사용자 행동이 느리면 훨씬 큰 사용자 규모를 뜻한다.

## 페이지/화면 단위 해석

프록시나 API 게이트웨이 앞단에서는 사용자 수보다 "페이지 1회 로드가 몇 개 API 요청을 만드는가"가 더 현실적일 때가 많다.

### 가정 D: 페이지 1회 로드에 API 요청 `10개`

| RPS | 초당 페이지 로드 | 분당 페이지 로드 |
| --- | ---: | ---: |
| `300` | `30` | `1,800` |
| `600` | `60` | `3,600` |
| `900` | `90` | `5,400` |
| `1200` | `120` | `7,200` |

해석:

- 이번 저장소의 프록시 벤치마크처럼 가벼운 `GET` 요청은 "서비스 전체 규모"보다 "특정 엔드포인트 또는 앞단 계층의 피크 요청률"로 읽는 편이 맞다.

## 응답시간을 포함한 동시 처리량 해석

같은 `RPS`라도 평균 응답시간이 길수록 시스템이 붙들고 있는 동시 작업 수는 커진다.

| RPS | 평균 `50ms` | 평균 `100ms` | 평균 `500ms` |
| --- | ---: | ---: | ---: |
| `300` | `15` | `30` | `150` |
| `600` | `30` | `60` | `300` |
| `900` | `45` | `90` | `450` |
| `1200` | `60` | `120` | `600` |

해석:

- 같은 `1200 RPS`라도 평균 응답시간이 `100ms`면 동시 처리 요청 약 `120개`, `500ms`면 약 `600개`다.
- 따라서 `RPS`만 보면 가벼워 보여도, 느린 비즈니스 로직에서는 훨씬 더 무거운 시스템 부하다.

## 현실적인 시나리오별 해석

아래 해석은 모두 **일반적 추론**이다. 실제 서비스 구조와 요청 비용에 따라 달라진다.

### 1. B2B SaaS의 핵심 읽기 API

| RPS | 현실적 해석 |
| --- | --- |
| `300` | 로그인 사용자 수천~수만 명 제품에서 핵심 조회 API의 피크 부하로 충분히 현실적이다. |
| `600` | 중형 제품의 핵심 대시보드/목록 API 피크 트래픽 수준으로 볼 수 있다. |
| `900~1200` | 단일 읽기 API 기준으로는 꽤 높은 편이며, 폴링이나 위젯 fan-out이 있는 제품에서 가능하다. |

### 2. 커머스/콘텐츠 서비스의 앞단 프록시

| RPS | 현실적 해석 |
| --- | --- |
| `300` | 중소형 서비스 피크 시간대 전체 앞단 트래픽으로 볼 수 있다. |
| `600~900` | 중형 서비스의 메인 프론트도어 또는 특정 이벤트 시간대 일부 구간에 해당할 수 있다. |
| `1200` | 대형 서비스라고 단정할 수는 없지만, 운영에서 무시할 수 없는 수준의 요청률이다. |

### 3. 내부 마이크로서비스 또는 fan-out 호출

| RPS | 현실적 해석 |
| --- | --- |
| `300` | 충분히 흔한 수준이다. |
| `600~1200` | 외부 사용자 수가 크지 않아도 aggregator나 내부 fan-out 때문에 쉽게 도달할 수 있다. |

### 4. 무거운 쓰기/주문/결제 API

| RPS | 현실적 해석 |
| --- | --- |
| `300` | 이미 꽤 높은 편이다. |
| `600` | DB 락, 트랜잭션, 외부 의존성이 있으면 상당한 부하일 수 있다. |
| `900~1200` | 프록시가 버티더라도 애플리케이션과 데이터 계층이 병목이 되기 쉽다. |

## 이 문서를 읽는 방법

실무에서는 아래 순서로 읽는 것이 안전하다.

1. 이 `RPS`가 서비스 전체 평균인지, 특정 API인지, 피크 구간인지 구분한다.
2. 요청이 가벼운 읽기인지, 무거운 쓰기인지 구분한다.
3. 사용자 행동 가정(`10초/30초/60초`) 중 어떤 시나리오와 더 가까운지 고른다.
4. 응답시간을 곱해 동시 처리량까지 같이 본다.
5. 그 다음에야 "대략 어느 정도 규모냐"를 말한다.

## 이번 벤치마크 문맥에 대한 적용

이번 저장소에서 측정한 `/api/info` 같은 가벼운 `GET` 요청은 아래처럼 읽는 편이 정확하다.

- `300~1200 RPS`는 "서비스 전체 평균 규모"보다 "특정 프록시 경로의 피크 요청률"에 가깝다.
- 프록시 계층 기준으로는 현실적인 비교 범위다.
- 하지만 이 수치만으로 곧바로 애플리케이션 전체 운영 가능 규모를 단정하면 안 된다.
- 실제 서비스 용량 판단에는 뒤쪽 앱 로직, DB, 캐시, 외부 API 비용을 따로 봐야 한다.

## 참고 자료

### 1. k6 constant-arrival-rate

- 링크: https://grafana.com/docs/k6/latest/using-k6/scenarios/executors/constant-arrival-rate/
- 근거로 사용한 내용:
  - 요청률 기반 테스트는 `rate`와 `timeUnit`으로 초당 시작 요청 수를 직접 정의한다.
  - 즉 `RPS`는 사용자 수가 아니라 시스템에 들어오는 요청 도착률이다.

### 2. k6 ramping-arrival-rate

- 링크: https://grafana.com/docs/k6/latest/using-k6/scenarios/executors/ramping-arrival-rate/
- 근거로 사용한 내용:
  - open-model 부하는 응답시간과 무관하게 지정한 속도로 iteration을 시작한다.
  - 따라서 `RPS` 해석은 "초당 몇 명의 사용자가 있다"가 아니라 "초당 몇 개 요청이 유입된다"가 기준이다.

### 3. k6 concurrent users guide

- 링크: https://grafana.com/docs/k6/latest/testing-guides/calculate-concurrent-users/
- 근거로 사용한 내용:
  - 사용자 수나 동시 활성 규모는 별도 행동 데이터와 세션 길이 가정이 있어야 계산할 수 있다.
  - 이 문서의 사용자 환산표는 그 점을 반영한 추정치다.

### 4. Cloudflare Workers pricing

- 링크: https://developers.cloudflare.com/workers/platform/pricing/
- 근거로 사용한 내용:
  - 월 `15M` 요청 예시를 통해 "월간 요청량"과 운영 트래픽 규모를 직관적으로 비교할 수 있다.
  - 본 문서의 `RPS -> 하루/월 요청량` 환산의 현실 감각 보조 자료로 사용했다.

### 5. Cloudflare Workers Logs

- 링크: https://developers.cloudflare.com/workers/observability/logs/workers-logs/
- 근거로 사용한 내용:
  - 월 `1B` 요청 예시가 있어 `300~1200 RPS` 지속 부하가 어느 정도 월간 트래픽인지 감을 잡는 참고점으로 썼다.

### 6. Firebase Cloud Messaging scale guide

- 링크: https://firebase.google.com/docs/cloud-messaging/scale-fcm
- 근거로 사용한 내용:
  - 대규모 시스템은 `RPS/QPS`와 토큰 버킷 단위로 요청률을 관리한다는 점을 보여준다.
  - 따라서 요청률 자체가 운영 규모와 연결되는 공통 언어라는 점을 보강하는 참고 자료로 사용했다.
